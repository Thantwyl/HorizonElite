FLIGHT BOOKING BACKEND API DOCUMENTATION
========================================

Project: Flight Booking Backend
API Version: 1.0.0
Default Base URL: http://localhost:3000
Swagger UI: http://localhost:3000/api-docs

Last updated: 2026-06-22


INSTALLATION
------------

1. Install Node.js
   Recommended: Node.js 18 or newer

2. Install PostgreSQL
   This backend uses PostgreSQL through the pg package.

3. Install project dependencies

   npm install

4. Run the database schema in your PostgreSQL database

   database/schema.sql


HOW TO START
------------

Development mode:

   npm run dev

Production/simple start:

   npm start

After starting, open:

   http://localhost:3000

Swagger UI:

   http://localhost:3000/api-docs

QUICK FRONTEND NOTES
--------------------

1. JWT token
   - Login returns the token.
   - Protected endpoints require:

     Authorization: Bearer <token>

2. Protected endpoints
   - GET /api/auth/profile
   - POST /api/flights/search

3. Flight search saves results to database
   - POST /api/flights/search returns both flight_search_id and flight results in one call.
   - Each result includes a flight_result_id linking to the saved DB record.
   - No separate Duffel API call needed.

4. Retrieve saved results
   - GET /api/results/:flight_search_id returns all saved offers for a given search.

5. Passenger Management
   - POST /api/passengers creates individual passenger records.
   - Save the passenger_id from each response.
   - Create one passenger record per traveler.

6. Booking Creation
   - POST /api/bookings creates a booking with selected flight and passengers.
   - Pass an array of passenger_ids when creating booking.
   - Response includes pnr_reference (unique 6-char booking code).
   - Booking starts in PENDING_PAYMENT status.

7. Payment Processing
   - POST /api/payment/create creates a payment record for the booking.
   - POST /api/payment/charge processes the actual payment through Omise gateway.
   - After successful payment, booking status updates to PAID_UNTICKETED.

8. Complete booking flow
   - Register → Login → Select Flight → Create Passengers → Create Booking → Process Payment


GENERAL NOTES
-------------

1. All request and response bodies use JSON.
2. Send this header for JSON requests:

   Content-Type: application/json

3. Protected endpoints require a JWT access token from the login API.
4. Send the JWT token using this header:

   Authorization: Bearer <token>

5. Common auth errors:

   401 Access Denied
   {
       "message": "Access Denied"
   }

   401 Invalid Token
   {
       "message": "Invalid Token"
   }

6. Common server error:

   500 Internal Server Error
   {
       "error": "Error message"
   }

FLIGHT SEARCH APIS
============================================================

1. Search and Create Flight Search
----------------------------------

Method:
POST

Endpoint:
/api/flights/search

Auth Required:
Yes

Headers:

Authorization: Bearer <jwt_token>
Content-Type: application/json

Description:
Creates a flight search record for the logged in user, stores one or more search segments,
and returns flight offers from the Duffel API in a single request.

Request Body:

{
    "trip_type": "ONE_WAY",
    "adult_passenger_count": 1,
    "child_passenger_count": 0,
    "infant_passenger_count": 0,
    "cabin_class": "economy",
    "preferred_airline": "Thai Airways",
    "direct_flight_only": false,
    "promo_fare_type": null,
    "currency_code": "USD",
    "segments": [
        {
            "origin_airport_code": "BKK",
            "destination_airport_code": "SIN",
            "departure_date": "2026-07-10"
        }
    ]
}

Required Fields:
- segments must exist and contain at least one segment.
- Each segment should include:
  - origin_airport_code
  - destination_airport_code
  - departure_date

Database Required Fields:
- trip_type
- adult_passenger_count
- child_passenger_count
- infant_passenger_count
- cabin_class
- currency_code

Allowed trip_type Values:
- ONE_WAY
- ROUND_TRIP
- MULTI_CITY

Notes:
- This endpoint uses the logged in user's email from the JWT token.
- The service inserts data into flight_searches and flight_search_segments.
- The endpoint also queries the Duffel API using the first segment's origin and destination.
- If Duffel API fails, the search is still created but results will be empty.

Success Response:
Status: 201

{
    "message": "Flight search created successfully",
    "flight_search_id": "uuid-value",
    "results": [
        {
            "flight_result_id": "uuid-value",
            "flight_offer_id": "off_0000000000000000",
            "airline_name": "Singapore Airlines",
            "airline_code": "SQ",
            "total_price": "120.00",
            "currency_code": "USD",
            "departure_airport": "BKK",
            "arrival_airport": "SIN",
            "departure_datetime": "2026-07-10T09:00:00",
            "arrival_datetime": "2026-07-10T12:30:00",
            "total_stop_count": 0,
            "cabin_class": "economy"
        }
    ]
}

Notes:
- Each result object includes flight_result_id — the UUID of the saved row in flight_results.
- Use this flight_result_id when calling POST /api/selected-flights.

Error Responses:

Status: 400
{
    "message": "At least one segment is required"
}

Status: 401
{
    "message": "Access Denied"
}

Status: 401
{
    "message": "Invalid Token"
}

============================================================
FRONTEND INTEGRATION FLOW
============================================================

Typical user flow:

1. Register user:
   POST /api/auth/register

2. Login user:
   POST /api/auth/login

3. Store token from login response.

4. Search flights:
   POST /api/flights/search
   Header: Authorization: Bearer <token>
   Response includes flight_search_id and results array.
   Each result includes a flight_result_id.

5. (Optional) Retrieve saved results later:
   GET /api/results/:flight_search_id
   Returns all saved offers with segment details.

6. Select one returned flight:
   POST /api/selected-flights
   Use flight_result_id from step 4 or 5.

7. Get logged in profile when needed:
   GET /api/auth/profile
   Header: Authorization: Bearer <token>

BOOKING APIS
============================================================

1. Create Booking
-----------------

Method:
POST

Endpoint:
/api/bookings

Auth Required:
No

Headers:

Content-Type: application/json

Description:
Creates a new booking with selected flight and passengers. Generates a 6-character PNR (Passenger Name Record).
The booking starts in PENDING_PAYMENT status.

Request Body:

{
    "user_email_address": "john@example.com",
    "selected_flight_id": "e7144f32-0885-4f44-93b9-bbfef6186d6a",
    "passenger_ids": [
        "uuid-passenger-1",
        "uuid-passenger-2"
    ],
    "total_payment_amount": 240.00,
    "currency_code": "USD",
    "trip_type": "ONE_WAY",
    "cabin_class": "ECONOMY",
    "fare_brand_id": "fare_brand_001"
}

Required Fields:
- user_email_address
- selected_flight_id (UUID)
- passenger_ids (array of passenger UUIDs, at least 1)
- total_payment_amount (decimal, > 0)
- currency_code (e.g., USD, THB, SGD)
- trip_type (ONE_WAY, ROUND_TRIP, MULTI_CITY)
- cabin_class (ECONOMY, PREMIUM_ECONOMY, BUSINESS, FIRST_CLASS)

Optional Fields:
- fare_brand_id

Notes:
- A unique PNR is automatically generated for the booking.
- Booking status is set to PENDING_PAYMENT by default.
- Ticketing status is set to UNTICKETED by default.

Success Response:
Status: 201

{
    "message": "Booking created successfully",
    "booking": {
        "pnr_reference": "ABC123",
        "user_email_address": "john@example.com",
        "selected_flight_id": "e7144f32-0885-4f44-93b9-bbfef6186d6a",
        "booking_status": "PENDING_PAYMENT",
        "ticketing_status": "UNTICKETED",
        "total_payment_amount": "240.00",
        "currency_code": "USD",
        "booking_date_timestamp": "2026-06-22T10:45:00.000Z",
        "booking_created_at": "2026-06-22T10:45:00.000Z",
        "booking_updated_at": "2026-06-22T10:45:00.000Z",
        "passengers": [
            {
                "passenger_id": "uuid-passenger-1",
                "pi_first_name": "John",
                "pi_last_name": "Doe",
                "pi_passenger_type_code": "ADT"
            },
            {
                "passenger_id": "uuid-passenger-2",
                "pi_first_name": "Jane",
                "pi_last_name": "Doe",
                "pi_passenger_type_code": "ADT"
            }
        ]
    }
}

Booking Status Values:
- PENDING_PAYMENT: Payment not yet received
- PAID_UNTICKETED: Payment received, waiting for ticketing
- TICKETED: Tickets issued
- CANCELLED: Booking cancelled
- REFUNDED: Refund processed

Ticketing Status Values:
- UNTICKETED: No tickets issued yet
- TICKETED: Tickets have been issued
- FAILED: Ticketing failed

Error Response:

Status: 500
{
    "error": "Error message"
}

}

Status: 500
{
    "error": "Error message"
}


2. Create Payment Record
------------------------

Method:
POST

Endpoint:
/api/payment/create

Auth Required:
No

Headers:

Content-Type: application/json

Description:
Creates a new payment record for a booking. This initiates the payment process but does not charge yet.

Request Body:

{
    "pnr_reference": "ABC123",
    "user_email_address": "john@example.com",
    "payment_method": "CREDIT_CARD",
    "payment_region": "ASIA",
    "currency_code": "USD",
    "total_payment_amount": 240.00,
    "payment_token": "tokn_test_123456"
}

Required Fields:
- pnr_reference (6-character booking reference)
- user_email_address
- payment_method (CREDIT_CARD, DEBIT_CARD, BANK_TRANSFER, etc.)
- payment_region (ASIA, EUROPE, AMERICAS, etc.)
- currency_code
- total_payment_amount (decimal, > 0)

Optional Fields:
- payment_token (from payment gateway)

Notes:
- Payment status is set to PENDING by default.
- The payment record links to the booking via PNR reference.

Success Response:
Status: 201

{
    "message": "Payment created",
    "payment": {
        "payment_id": "uuid-value",
        "pnr_reference": "ABC123",
        "user_email_address": "john@example.com",
        "payment_method": "CREDIT_CARD",
        "payment_region": "ASIA",
        "currency_code": "USD",
        "total_payment_amount": "240.00",
        "payment_token": "tokn_test_123456",
        "three_ds_status": null,
        "payment_status_code": "PENDING",
        "payment_timestamp": "2026-06-22T10:55:00.000Z"
    }
}

Payment Status Codes:
- PENDING: Payment initiated, awaiting processing
- PAID: Payment successfully processed
- FAILED: Payment transaction failed
- REFUNDED: Payment refunded
- VOIDED: Payment voided

3DS Status Values:
- AUTHENTICATED: 3D Secure authentication passed
- ATTEMPTED: 3D Secure attempted
- FAILED: 3D Secure authentication failed

Error Response:

Status: 500
{
    "error": "Error message"
}


3. Charge Payment (Process Transaction)
----------------------------------------

Method:
POST

Endpoint:
/api/payment/charge

Auth Required:
No

Headers:

Content-Type: application/json

Description:
Processes the payment transaction through the Omise payment gateway. This actually charges the payment method.

Request Body:

{
    "payment_id": "uuid-value",
    "pnr_reference": "ABC123",
    "payment_token": "tokn_test_123456",
    "amount": 240.00,
    "currency": "USD",
    "description": "Flight booking ABC123",
    "return_uri": "https://frontend.example.com/payment/result",
    "metadata": {
        "booking_reference": "ABC123",
        "user_email": "john@example.com"
    }
}

Required Fields:
- payment_id (UUID from create payment)
- pnr_reference
- payment_token
- amount (in smallest currency unit, e.g., cents for USD)
- currency (ISO 4217 code)

Optional Fields:
- description
- return_uri (for 3D Secure redirect)
- metadata (custom key-value pairs)

Success Response:
Status: 200

{
    "payment_id": "uuid-value",
    "pnr_reference": "ABC123",
    "payment_status_code": "PAID",
    "gateway_transaction_reference": "chrg_test_123456",
    "three_ds_status": "AUTHENTICATED",
    "amount": "240.00",
    "currency": "USD",
    "payment_timestamp": "2026-06-22T11:00:00.000Z",
    "message": "Payment processed successfully"
}

3D Secure Flow:
- If 3D Secure is required, the response will include a redirect URL.
- After user completes 3D Secure challenge, payment is finalized.
- three_ds_status will show AUTHENTICATED or ATTEMPTED status.

Error Response:

Status: 500
{
    "error": "Error message"
}

Status: 400
{
    "error": "Invalid payment token"
}


============================================================
TYPICAL BOOKING AND PAYMENT FLOW
============================================================

Complete end-to-end booking with payment:

1. Register and Login (see Authentication APIs)

2. Search and select flights (see Flight Search APIs)

3. Create passengers:
   POST /api/passengers
   Repeat for each passenger. Save passenger_ids.

4. Create booking:
   POST /api/bookings
   Use selected_flight_id and passenger_ids.
   Save pnr_reference (booking reference).

5. Create payment record:
   POST /api/payment/create
   Use pnr_reference and booking amount.

6. Test payment gateway (optional):
   GET /api/payment/test

7. Process payment:
   POST /api/payment/charge
   Complete the transaction.
   Booking status updates to PAID_UNTICKETED upon success.

8. Booking is ready for ticketing:
   booking_status = PAID_UNTICKETED
   Tickets can now be issued via airline system.
